# LibrePhotos

LibrePhotos is the OSS self-hosted Google Photos alternative with **smart organization**: face recognition groups faces, object detection makes "show me photos with dogs" searchable, reverse geocoding puts photos on a map, and auto-albums ("Thursday in Berlin") group by time + location + people.

Evolved from the **Ownphotos** project; repo = `LibrePhotos/librephotos`. **Heavy on ML** — expect CPU/RAM usage during scans.

Features:

- **Face recognition** — label a few faces, classifier learns the rest
- **Object detection** — "beach," "dog," "car," "food" — text-searchable
- **Places365** — scene classification (indoor/outdoor/urban/nature)
- **Reverse geocoding** — map view of where photos were taken (needs Mapbox key for free tier, or other providers)
- **Smart albums** — auto-generated by event + place + people
- **Manual albums** — your own organization
- **Face verification** — label-then-train loop for accuracy
- **Duplicate detection**
- **Share** albums/photos publicly or with users
- **Multi-user** — each user has their own library
- **Timeline view** — by year/month/day
- **Search** — by object, face, date, place, or free text
- **Thumbnails + HEIC/RAW support**

## Architecture in one minute

Multi-container:

1. **backend** (Django + face_recognition + PyTorch + Places365) — the heavy worker
2. **frontend** (React SPA)
3. **Postgres** — metadata (users, photos, faces, albums)
4. **Redis** — task queue (Celery / RQ), cache
5. **proxy** (nginx) — routes frontend + API + serves thumbnails

- **GPU is optional but helps** — face + object detection on CPU is 10-20x slower
- **First scan of a large library** can take hours → days on CPU
- **RAM**: 8+ GB recommended; 16+ for large libraries
- **Disk**: original photos + generated thumbnails (typically 10-20% extra)

## Install via Docker Compose

Take the upstream compose from the repo (it's multi-service and changes over releases). Key pattern:

```yaml
services:
  db:
    image: postgres:13-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: lp_user
      POSTGRES_PASSWORD: <strong>
      POSTGRES_DB: lp_db
    volumes:
      - ./data/db:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    restart: unless-stopped

  backend:
    image: reallibrephotos/librephotos:latest   # pin specific version in prod
    restart: unless-stopped
    depends_on: [db, redis]
    environment:
      ADMIN_EMAIL: admin@example.com
      ADMIN_USERNAME: admin
      ADMIN_PASSWORD: <strong>
      DB_USER: lp_user
      DB_PASS: <strong>
      DB_NAME: lp_db
      DB_HOST: db
      REDIS_HOST: redis
      REDIS_PORT: 6379
      MAPBOX_API_KEY: <optional>
      SECRET_KEY: <64-random-chars>
      BACKEND_HOST: backend
      ALLOWED_HOSTS: photos.example.com
      TIME_ZONE: America/Los_Angeles
      HEAVYWEIGHT_PROCESS: 1      # number of parallel ML workers
    volumes:
      - /mnt/photos:/data/photos:ro
      - ./data/protected_media:/protected_media
      - ./data/data_models:/data_models
      - ./data/logs:/logs
      - ./data/cache:/root/.cache

  frontend:
    image: reallibrephotos/librephotos-frontend:latest
    restart: unless-stopped

  proxy:
    image: reallibrephotos/librephotos-proxy:latest
    restart: unless-stopped
    depends_on: [backend, frontend]
    ports:
      - "3000:80"
    volumes:
      - /mnt/photos:/data/photos:ro
      - ./data/protected_media:/protected_media
```

Front with Caddy/Traefik for TLS. Browse `https://photos.example.com` → log in as admin.

## First boot

1. Log in as admin
2. **Library → Scan** — starts indexing (watch logs: `docker logs -f librephotos-backend`)
3. Wait. 10k photos on CPU ≈ 2-8 hours. GPU cuts to minutes.
4. After initial scan: **People** page — shows detected faces grouped by cluster. Click a cluster, name it → classifier re-trains overnight.
5. **Search bar**: "dog", "beach", "sunset", or people's names.
6. **Map** view: photos with EXIF GPS appear.
7. Settings → add users (family members).

## Data & config layout

- `data/db` — Postgres
- `data/protected_media` — generated thumbnails + derivatives (grows with library size)
- `data/data_models` — downloaded ML models (~2-4 GB for face + object detection)
- `data/cache` — HEIC conversion cache
- `/data/photos` — your originals (**never written to**; mount :ro for safety)

## Backup

```sh
# DB (metadata, faces, albums — CRITICAL)
docker exec lp-db pg_dump -U lp_user lp_db | gzip > lp-db-$(date +%F).sql.gz

# Thumbnails (regenerable but slow)
tar czf lp-media-$(date +%F).tgz data/protected_media/

# Your originals — back up separately via your existing photo backup strategy
# (LibrePhotos does NOT modify originals)
```

## Upgrade

1. Releases: <https://github.com/LibrePhotos/librephotos/releases>. Moderate activity.
2. **Read release notes** — migrations happen; occasional breaking changes.
3. Bump backend + frontend + proxy image tags → `docker compose pull && docker compose up -d`.
4. Back up DB first.
5. ML model versions may change → first scan after upgrade re-processes some data.

## Gotchas

- **Initial scan is SLOW on CPU** — plan for hours/days for large libraries (10k+). Run over a weekend. Monitor `docker stats` — CPU pegged at 100% is normal.
- **RAM usage** during scan can spike to 8+ GB; OOM-kills of the backend halt scans. Use a server with memory, or reduce `HEAVYWEIGHT_PROCESS` to 1.
- **GPU acceleration**: add `deploy.resources.reservations.devices` with NVIDIA runtime + use `librephotos-gpu` image tag. Check upstream docs for current state.
- **Face recognition accuracy** — depends on training data. Give it 5-10 faces per person initially; accuracy improves. Mislabeled faces corrupt clusters.
- **Child vs adult face** clustering drifts as kids grow; expect to re-cluster annually.
- **Library read-only mount** is a safety feature — LibrePhotos doesn't overwrite your originals. Good default.
- **HEIC / HEIF** (iPhone) — supported; requires conversion step which uses disk + CPU. Shows up as a slightly slow first view.
- **RAW files** — supported but take longer to process; you get a preview, not full editing.
- **Mapbox key** — for map view, free tier OK for single user (50k/mo geocodes); add a key or switch to Nominatim (OSM).
- **No sync app** — unlike Immich / PhotoPrism, no first-class mobile sync. You copy photos to the mount point via rsync / Syncthing / Nextcloud sync. This is LibrePhotos' main UX weakness.
- **Sharing** is within-instance; no public share by design for most setups.
- **Django ALLOWED_HOSTS** must match the hostname you access via, or Django refuses.
- **CSRF** — set `CSRF_TRUSTED_ORIGINS` if using an unusual reverse proxy setup.
- **Multi-user isolation** — each user has their own photos; admin sees all.
- **Duplicate detection** relies on hash; very similar but not identical photos (e.g., different resolutions) may not dedupe.
- **Project cadence** — active but slower than Immich. Expect rough edges + occasional breaking releases. Read issues before upgrading.
- **Performance tuning**: upstream wiki has guides on `HEAVYWEIGHT_PROCESS`, Postgres tuning, Redis persistence.
- **vs Immich**: Immich has better mobile apps, sync, UI polish, active development. LibrePhotos has better face/object ML out of the box but weaker sync. Many people try Immich first.
- **vs PhotoPrism**: PhotoPrism (Go) is faster + smaller deps; ML is more limited but clean.
- **License** — MIT. Permissive.
- **Alternatives worth knowing:**
  - **Immich** — modern, active, great mobile apps (separate recipe; likely higher in the queue)
  - **PhotoPrism** — Go-based, fast, clean UX (separate recipe)
  - **PhotoStructure** — commercial + free; desktop app; pragmatic
  - **Piwigo** — classic PHP photo gallery; no ML (separate recipe)
  - **Lychee** — album-focused gallery
  - **Damselfly** — another OSS face/photo manager
  - **Choose LibrePhotos if:** you want face + object ML now on a server you already have, and don't prioritize mobile sync.
  - **Choose Immich if:** mobile sync + active dev + polished UX matter.
  - **Choose PhotoPrism if:** you want lean + fast + Go single-binary simplicity.
