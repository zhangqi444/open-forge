# Dependency-Track

OWASP Flagship intelligent component analysis platform for software supply chain risk management. Consumes Software Bill of Materials (SBOM in CycloneDX format) to identify known vulnerabilities, out-of-date components, and license risks across every application in your portfolio. API-first design, ideal for CI/CD integration. Upstream: <https://github.com/DependencyTrack/dependency-track>. Docs: <https://docs.dependencytrack.org/>.

## Compatible install methods

| Method | When to use |
|---|---|
| Docker Compose (bundled, recommended for eval) | Single-container quickstart |
| Docker Compose (API server + frontend) | Production; separate scalable services |
| Helm (Kubernetes) | K8s production |

## Inputs to collect

| Phase | Prompt | Notes |
|---|---|---|
| preflight | "Bundled (single container) or separated (API + frontend)?" | Bundled is easiest; separated is recommended for production |

## Docker Compose (separated, recommended for production)

```yaml
version: "3.9"

volumes:
  dependency-track:

services:
  dtrack-apiserver:
    image: dependencytrack/apiserver:latest
    restart: unless-stopped
    environment:
      ALPINE_DATABASE_MODE: internal   # embedded H2; use external Postgres for production
    ports:
      - "8081:8080"
    volumes:
      - dependency-track:/data
    mem_limit: 4096m
    mem_reservation: 2048m

  dtrack-frontend:
    image: dependencytrack/frontend:latest
    restart: unless-stopped
    depends_on:
      - dtrack-apiserver
    environment:
      API_BASE_URL: http://localhost:8081
    ports:
      - "8080:8080"
```

- UI: http://localhost:8080 (default admin: `admin` / `admin` — change immediately)
- API: http://localhost:8081

## PostgreSQL backend (production)

Replace `ALPINE_DATABASE_MODE: internal` with:

```yaml
ALPINE_DATABASE_MODE: external
ALPINE_DATABASE_URL: jdbc:postgresql://db:5432/dtrack
ALPINE_DATABASE_DRIVER: org.postgresql.Driver
ALPINE_DATABASE_USERNAME: dtrack
ALPINE_DATABASE_PASSWORD: changeme
```

## CI/CD integration

Upload SBOMs generated by tools like Syft, cdxgen, or Trivy:

```bash
# Generate SBOM with Syft
syft myapp:latest -o cyclonedx-json > sbom.json

# Upload to Dependency-Track
curl -X PUT http://localhost:8081/api/v1/bom \
  -H "X-Api-Key: <your-api-key>" \
  -F "projectName=myapp" \
  -F "projectVersion=1.0" \
  -F "bom=@sbom.json"
```

## Software-layer concerns

- UI port: `8080`, API port: `8080` (bundled) or `8081` (separated)
- Default credentials: `admin` / `admin` — **change on first login**
- Vulnerability intel sources: NVD, GitHub Advisories, OSS Index, Snyk, Trivy — configure in Administration
- Policy engine: define license policies and vulnerability severity thresholds per project
- Notifications: Slack, email, Teams, PagerDuty webhook support
- `mem_limit: 4096m` recommended — Java process is memory-hungry

## Upgrade procedure

1. Pull new images: `docker compose pull`
2. Restart: `docker compose up -d`
3. DB schema migrations run automatically on API server startup

## Gotchas

- **Change default admin password immediately** — `admin`/`admin` is well-known
- Internal H2 database is fine for eval; PostgreSQL strongly recommended for production (H2 has no backup tooling)
- API server needs ≥ 4 GB RAM; reduce `mem_limit` at your own risk
- `API_BASE_URL` in the frontend container must be reachable from the **browser** (not just the Docker network)
